# 阶段 5 · 综合项目 capstone：多工具代码库问答 agent 🏁

把前 4 阶段全部串起来，做一个**迷你版 Claude Code**：给 agent 几个操作文件的工具，
它就能**自主探索一个真实目录**来回答你的问题——先列目录 / 搜关键字，定位到相关文件，
再读文件细节，最后基于内容作答。整个「探索几步、先看哪、再看哪」由**模型自己规划**。

练手对象是 `sandbox/`（一个示例订单服务）。你的 agent 要能回答类似：
> "这个服务部署到生产用的端口是多少？定义在哪个文件里？"

## 三个「真实 agent 必备」的新概念（这一阶段的重点）
前 4 阶段的 agent 是"玩具"，因为它假设一切顺利。真实 agent 要处理"不顺利"：

1. **工具会失败 → 错误要回传给模型，不能让程序崩。**
   文件不存在、路径写错……工具要返回一句「错误：……」的字符串，而不是抛异常。
   模型看到这句错误，**下一轮还能自己换个法子重试**。`execute()` 再用 try/except 兜一层底，
   保证任何一个工具炸了都不会掀翻整个循环。这是玩具和生产 agent 的分水岭。

2. **循环要有护栏 → `MAX_STEPS`。**
   阶段 3 用的是 `while True`，万一模型钻进死胡同反复调工具，会把配额烧光。
   真实 agent 一定有步数上限：到顶就停、返回目前的结果。阶段 3 说这留到"阶段 6"，现在你亲手加上。

3. **用系统提示词(system)塑造行为，用多工具扩展能力。**
   阶段 3 只有 1 个工具、任务也直白，不写 `system` 也能跑。可工具一多（3 个）、任务变成
   "自主探索"，模型就容易**瞎猜文件名、把"先搜再读"的顺序搞反**——`system` 就是你交给它的
   工作策略，没有它行为会飘（每轮请求都要带上）。同时工具一多，模型要靠每个工具的
   `description` 自己选调哪个（所以描述变成关键），`execute` 也要按 `name` 把调用**路由**到对应函数。

## 系统提示词该写什么（`SYSTEM_PROMPT`）
它是模型的"岗位说明书"。自己组织语言，至少讲清楚这几点：
- **身份**：你是一个代码库问答助手，通过调用工具来探索项目、回答关于代码的问题。
- **工作方式**：先用 `list_dir` / `search_files` 找到相关文件，再用 `read_file` 看细节，
  别凭空猜文件名或内容。
- **答问要求**：基于文件里真实读到的内容作答；**下结论前先用 `read_file` 打开文件确认，
  不要只凭一条搜索片段就作答**；说清楚答案在哪个文件里；读不到就如实说不知道。
- **范围**：项目根目录是 `sandbox`。

## 通关标准
**离线段（纯 Python，不花钱）**
- `list_dir` / `read_file` / `search_files` 都能正常工作，且**坏输入返回「错误：」而不抛异常**；
  `read_file` 会截断超长内容。
- `TOOLS` 三份 schema 齐全、名字/参数名和函数对得上、description 非空。
- `execute` 分发正确；未知工具、工具抛异常都返回「错误：」。
- **护栏生效**：用一个"永远要工具的假模型"测试，你的循环能在 `MAX_STEPS` 轮内停下（不无限转），
  且每轮都带上了 `system` 和 `tools`。

**在线段（真调一次模型）**
- 模型请求 **≥ 2 次**（多步）、用了**发现类工具**(list_dir/search_files) + **read_file**、
  最终回答给出正确端口号（藏在 `sandbox/config/settings.py` 里）。
