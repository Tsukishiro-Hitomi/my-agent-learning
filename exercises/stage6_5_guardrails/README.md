# 阶段 6.5 · 安全护栏 Guardrails 🚦（动手练习）

agent 越自主，越可能干出**不可逆/有代价**的事：删文件、发邮件、花钱。
阶段 5 的 `MAX_STEPS` 是一种护栏（防跑飞）；这一关加更重要的一种——
**人在环 (human-in-the-loop)**：碰到危险操作，先停下来问人 `确定吗？(y/n)`。

```
模型想调 delete_file
        │
   危险工具吗？──否──▶ 直接执行
        │是
   问用户 (y/n)
    ├─ y → 执行
    └─ n → 不执行，把"被拒绝"当结果回给模型（它可以换个做法）
```

## 你要做的（就三样）
- **A** `DANGEROUS`：把"会改动/删除东西"的工具名放进这个集合（给你的工具里，哪些危险？）。
- **B** `confirm(name, args)`：危险操作前用 `_ask(...)` 问人，`y/是` 才返回 True。
- **C** `guarded_execute(name, args)`：危险的先 `confirm`，被拒就返回「错误：用户拒绝执行 …」
  （**返回字符串**，不是抛异常——让模型知道被拒、可以改做法）；安全的直接执行。

真正干活的工具（read/write/delete）、询问钩子 `_ask`、agent 循环，**都给你了**。写完 `./run.sh 6.5`。

## 关键点：被拒绝要"温柔地"告诉模型
拒绝时不要抛异常让程序崩，而是像工具出错一样**返回一句 tool_result**（"用户拒绝执行 X"）。
这样模型能读到、理解"这条路被人挡了"，从而换个做法或如实告诉用户——这正是阶段 5 学的
"工具错误回传给模型"的思想，用在了权限上。

## 通关标准
**离线（主体，不花钱）**：危险操作被拒 → 文件安然无恙；被批准 → 真的执行；安全工具**不问**直接放行；
`DANGEROUS` 配置合理；异常被兜住。（评测用替换 `_ask` 的方式模拟你按 y/n，不用真的手输。）
**在线**：让 agent 去删一个文件，但确认设成"自动拒绝"——文件**还在**，说明护栏挡住了模型。

自动化不等于放任。**在不可逆/有代价的动作上留一道人工闸门**，是让 agent 能安全上线的关键一环。
